Setup — Private Registry (GHCR) Scanning
-----------------------------------------
Source of truth: ghcr-pull-secret in {{ .Values.ghcrRegistryAuth.eso.sourceNamespace }}
  Keys: Username (GitHub username), Password (GHCR PAT, read:packages scope)

ESO syncs that secret into this namespace (for Trivy) and into every namespace
labeled glerp.io/private-registry=true (for pod pulls + future apps).
New GLerp apps need only: kubectl label namespace <ns> glerp.io/private-registry=true

STEP 1 — One-time cluster prerequisites (cluster-admin, run once per environment)
  # Grant ESO's controller SA read access to ghcr-pull-secret in the source namespace.
  # Replace <eso-namespace> and <eso-sa-name> with your ESO install values
  # (commonly: namespace=external-secrets, sa=external-secrets).
  kubectl create role eso-ghcr-reader \
    -n {{ .Values.ghcrRegistryAuth.eso.sourceNamespace }} \
    --verb=get,list,watch \
    --resource=secrets \
    --resource-name={{ .Values.ghcrRegistryAuth.secretName }} \
    --request-timeout=5s

  kubectl create rolebinding eso-ghcr-reader \
    -n {{ .Values.ghcrRegistryAuth.eso.sourceNamespace }} \
    --role=eso-ghcr-reader \
    --serviceaccount=<eso-namespace>:<eso-sa-name> \
    --request-timeout=5s

  # Create the ClusterSecretStore (Kubernetes provider, cross-namespace read).
  # Name must match ghcrRegistryAuth.eso.clusterSecretStoreName in your values.
  kubectl apply --request-timeout=5s -f - <<'EOF'
  apiVersion: external-secrets.io/v1beta1
  kind: ClusterSecretStore
  metadata:
    name: k8s-{{ .Values.ghcrRegistryAuth.eso.sourceNamespace }}
  spec:
    provider:
      kubernetes:
        remoteNamespace: {{ .Values.ghcrRegistryAuth.eso.sourceNamespace }}
        server:
          url: "https://kubernetes.default.svc"
          caProvider:
            type: ConfigMap
            name: kube-root-ca.crt
            key: ca.crt
            namespace: <eso-namespace>
        auth:
          serviceAccount:
            name: <eso-sa-name>
            namespace: <eso-namespace>
  EOF

  # Label this namespace so ESO's ClusterExternalSecret includes it:
  kubectl label namespace {{ .Release.Namespace }} \
    glerp.io/private-registry=true --request-timeout=5s

  # For production: repeat Step 1 with sourceNamespace pointing to the prod namespace
  # that holds ghcr-pull-secret, then set ghcrRegistryAuth.eso.sourceNamespace in
  # your prod values override.

STEP 2 — Enable in values override and upgrade
  # Add to your values override file:
  #
  #   ghcrRegistryAuth:
  #     enabled: true
  #     eso:
  #       enabled: true
  #       clusterSecretStoreName: "k8s-{{ .Values.ghcrRegistryAuth.eso.sourceNamespace }}"
  #       clusterExternalSecret:
  #         enabled: true             # spreads ghcr-pull-secret to all labeled namespaces
  #
  #   trivy-operator:
  #     trivyOperator:
  #       scanJobAdditionalEnvVars:
  #         - name: TRIVY_USERNAME
  #           valueFrom:
  #             secretKeyRef:
  #               name: {{ .Values.ghcrRegistryAuth.secretName }}
  #               key: {{ .Values.ghcrRegistryAuth.eso.usernameKey }}
  #         - name: TRIVY_PASSWORD
  #           valueFrom:
  #             secretKeyRef:
  #               name: {{ .Values.ghcrRegistryAuth.secretName }}
  #               key: {{ .Values.ghcrRegistryAuth.eso.passwordKey }}

  helm upgrade {{ .Release.Name }} green-llama/glerp-sec-audit \
    -n {{ .Release.Namespace }} -f <your-values-override.yaml> \
    --request-timeout=60s

  # Force re-scan of private images once credentials are in place:
  kubectl delete vulnerabilityreports -A --all --request-timeout=30s

Diagnose — registry.k8s.io TLS errors in Trivy logs
------------------------------------------------------
  DNS test confirmed registry.k8s.io resolves to its correct public IP (34.96.108.209).
  If Trivy still reports TLS cert mismatch for registry.k8s.io, the cause is NOT DNS.

  Most likely cause — Rancher System Default Registry:
    Rancher UI → Home → Global Settings → system-default-registry
    If set to an internal cache URL, Rancher prepends that host to ALL image references
    at the CRI level, including images pulled by Trivy scan pods.
    Fix: clear the setting to empty, OR ensure the internal cache's IngressRoute does
    NOT have the Google MFA middleware (registries authenticate with their own token
    protocol; web-based MFA breaks all automated clients including Trivy and containerd).

  Check the setting without Rancher UI:
    kubectl get settings.management.cattle.io system-default-registry \
      -o jsonpath='{.value}' --request-timeout=5s

  Temporary chart workaround (if the internal cert is from a trusted CA):
    Add to your values override under trivy-operator:
      trivy:
        insecureRegistries: "registry.k8s.io"

Debug — Trivy Metrics
---------------------

  kubectl exec -n {{ .Release.Namespace }} \
    deploy/{{ include "glerp-sec-audit.fullname" . }}-trivy-operator -- \
    wget -qO- http://localhost:8080/metrics 2>/dev/null \
    | grep "^trivy_cluster_compliance" | head -10

    deploy/{{ include "glerp-sec-audit.fullname" . }}-trivy-operator -- \
    wget -qO- http://localhost:8080/metrics 2>/dev/null \
    | grep "^trivy_image_vulnerabilities" | wc -l

  # Force re-scan when VulnerabilityReports are stale/empty
  kubectl delete vulnerabilityreports -A --all --request-timeout=30s
