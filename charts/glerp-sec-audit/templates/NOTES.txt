Setup — Private Registry (GHCR) Scanning
-----------------------------------------
Source of truth: {{ .Values.ghcrRegistryAuth.secretName }} in {{ .Values.ghcrRegistryAuth.eso.sourceNamespace }}
  Keys: {{ .Values.ghcrRegistryAuth.eso.usernameKey }} / {{ .Values.ghcrRegistryAuth.eso.passwordKey }}

ESO syncs that secret into this namespace (for Trivy) and into every namespace
labeled glerp.io/private-registry=true (for pod image pulls + future apps).
The chart manages the ClusterSecretStore and ExternalSecret resources automatically.
New GLerp apps need only: kubectl label namespace <ns> glerp.io/private-registry=true

STEP 1 — One-time RBAC grant (cluster-admin, run ONCE per environment)
  This is the only manual step. Grant ESO's controller SA permission to read
  {{ .Values.ghcrRegistryAuth.secretName }} from {{ .Values.ghcrRegistryAuth.eso.sourceNamespace }}.

  kubectl create role eso-ghcr-reader \
    -n {{ .Values.ghcrRegistryAuth.eso.sourceNamespace }} \
    --verb=get,list,watch \
    --resource=secrets \
    --resource-name={{ .Values.ghcrRegistryAuth.secretName }} \
    --request-timeout=5s

  kubectl create rolebinding eso-ghcr-reader \
    -n {{ .Values.ghcrRegistryAuth.eso.sourceNamespace }} \
    --role=eso-ghcr-reader \
    --serviceaccount={{ .Values.ghcrRegistryAuth.eso.controllerNamespace }}:{{ .Values.ghcrRegistryAuth.eso.controllerServiceAccount }} \
    --request-timeout=5s

  # Label namespaces that run private GHCR images (including this one for Trivy):
  kubectl label namespace {{ .Release.Namespace }} \
    glerp.io/private-registry=true --request-timeout=5s
  kubectl label namespace {{ .Values.ghcrRegistryAuth.eso.sourceNamespace }} \
    glerp.io/private-registry=true --request-timeout=5s

  For production: re-run Step 1 substituting the prod sourceNamespace value.
  No other environment-specific steps — the chart handles the rest.

Diagnose — registry.k8s.io TLS errors in Trivy logs
------------------------------------------------------
  Confirmed non-issues (already verified):
    - DNS resolves to public IP 34.96.108.209 (not intercepted)
    - system-default-registry is empty (Rancher not redirecting pulls)
    - No Traefik IngressRoute for registry.k8s.io

  Most likely remaining cause — Istio sidecar injection:
    If the {{ .Release.Namespace }} namespace has istio-injection=enabled,
    the Istio sidecar intercepts all outbound TCP from Trivy pods (including
    direct TLS pulls to external registries) and applies mTLS, causing cert errors.

    Check: kubectl get namespace {{ .Release.Namespace }} --show-labels
    Fix:   kubectl label namespace {{ .Release.Namespace }} istio-injection=disabled \
             --overwrite --request-timeout=5s
           # Restart Trivy operator pod to drop the sidecar:
           kubectl rollout restart -n {{ .Release.Namespace }} \
             deploy/{{ include "glerp-sec-audit.fullname" . }}-trivy-operator \
             --request-timeout=5s

  If Istio injection is not enabled and the error persists:
    The namespace likely inherited the label from a namespace template.
    Check for the annotation: kubectl get namespace {{ .Release.Namespace }} -o yaml
    Also check: kubectl get mutatingwebhookconfiguration istio-sidecar-injector \
      -o jsonpath='{.webhooks[0].namespaceSelector}' --request-timeout=5s

  Temporary chart workaround (skip TLS verification for registry.k8s.io only):
    Add to your values override under trivy-operator:
      trivy:
        insecureRegistries: "registry.k8s.io"
