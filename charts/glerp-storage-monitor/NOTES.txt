STEP 1 — REQUIRED: Enable 30-day historical data (one-time Rancher Monitoring change)
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

In Rancher UI → Apps → rancher-monitoring → Edit → paste into YAML:

  prometheus:
    prometheusSpec:
      remoteWrite:
        - url: http://{{ .Release.Name }}-victoriametrics.{{ .Values.namespace }}.svc:8428/api/v1/write
          writeRelabelConfigs:
            - sourceLabels: [__name__]
              regex: "node_filesystem.*|longhorn.*|minio.*|directpv.*|kube_customresource_directpv.*|kube_persistentvolume.*|kube_persistentvolumeclaim.*|glerp:.*|probe_.*|glerp_maintenance.*|trivy_.*|kube_bench_.*|zap_.*"
              action: keep

This VictoriaMetrics instance is the SHARED long-term store for ALL GLerp projects:
  - glerp-storage-monitor: storage/capacity metrics (90-day history)
  - glerp-monitoring: SLA uptime probe data (probe_.*) for compliance reports
  - Both use the Grafana datasource "GLerp — VictoriaMetrics" (UID: glerp-vm)

Without this step, historical trend panels and SLA compliance dashboards will show no data.

STEP 2 — Before MinIO dashboards: Enable public MinIO metrics (per-tenant)
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

For each customer's MinIO Tenant, add this environment variable to the Tenant spec:

  spec:
    env:
      - name: MINIO_PROMETHEUS_AUTH_TYPE
        value: "public"

Apply via kubectl patch (replace TENANT_NAME and NAMESPACE):

  kubectl patch tenant <TENANT_NAME> -n <NAMESPACE> \
    --type=json \
    -p '[{"op":"add","path":"/spec/env/-","value":{"name":"MINIO_PROMETHEUS_AUTH_TYPE","value":"public"}}]'

This enables unauthenticated cluster-internal Prometheus scraping. It is safe because
Prometheus runs inside the cluster — no credentials are stored in this chart.

After patching, the MinIO pods will restart automatically. Metrics will appear in
Prometheus within one scrape interval (~60s).

STEP 4 — Enable DirectPV physical drive capacity metrics (one-time)
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

DirectPV only exposes per-volume written bytes via its metrics endpoint — physical drive
capacity (e.g. 25 GiB per node) lives in the DirectPVDrive CRD. This step configures
kube-state-metrics to read those CRDs and expose three Prometheus metrics per node.

IMPORTANT: kube-state-metrics automatically prepends "kube_customresource_" to all custom
resource metric names. The actual metric names in Prometheus are:
  kube_customresource_directpv_drive_total_capacity_bytes
  kube_customresource_directpv_drive_allocated_capacity_bytes
  kube_customresource_directpv_drive_free_capacity_bytes

4a. Identify the kube-state-metrics ServiceAccount name on your cluster:

  kubectl get sa -n cattle-monitoring-system --request-timeout=5s | grep kube-state
  # Typically: rancher-monitoring-kube-state-metrics

4b. Grant kube-state-metrics read access to DirectPVDrive CRDs
    (replace the ServiceAccount name if different from the default):

  kubectl apply -f - <<'EOF'
  apiVersion: rbac.authorization.k8s.io/v1
  kind: ClusterRole
  metadata:
    name: kube-state-metrics-directpv
  rules:
    - apiGroups: ["directpv.min.io"]
      resources: ["directpvdrives"]
      verbs: ["get", "list", "watch"]
  ---
  apiVersion: rbac.authorization.k8s.io/v1
  kind: ClusterRoleBinding
  metadata:
    name: kube-state-metrics-directpv
  roleRef:
    apiGroup: rbac.authorization.k8s.io
    kind: ClusterRole
    name: kube-state-metrics-directpv
  subjects:
    - kind: ServiceAccount
      name: rancher-monitoring-kube-state-metrics
      namespace: cattle-monitoring-system
  EOF

4c. Add custom resource state to rancher-monitoring via Rancher UI
    (Apps → rancher-monitoring → Edit/Upgrade → Edit YAML):

  kube-state-metrics:
    customResourceState:
      enabled: true
      config:
        spec:
          resources:
            - groupVersionKind:
                group: directpv.min.io
                version: v1beta1
                kind: DirectPVDrive
              metrics:
                - name: directpv_drive_total_capacity_bytes
                  help: "Total physical capacity of DirectPV drive"
                  each:
                    type: Gauge
                    gauge:
                      path: [status, totalCapacity]
                  labelsFromPath:
                    node: [metadata, labels, "directpv.min.io/node"]
                    drive: [metadata, labels, "directpv.min.io/drive-name"]
                - name: directpv_drive_allocated_capacity_bytes
                  help: "Allocated capacity of DirectPV drive (sum of provisioned PVC sizes)"
                  each:
                    type: Gauge
                    gauge:
                      path: [status, allocatedCapacity]
                  labelsFromPath:
                    node: [metadata, labels, "directpv.min.io/node"]
                    drive: [metadata, labels, "directpv.min.io/drive-name"]
                - name: directpv_drive_free_capacity_bytes
                  help: "Free physical capacity of DirectPV drive"
                  each:
                    type: Gauge
                    gauge:
                      path: [status, freeCapacity]
                  labelsFromPath:
                    node: [metadata, labels, "directpv.min.io/node"]
                    drive: [metadata, labels, "directpv.min.io/drive-name"]

4d. Force restart kube-state-metrics after the rancher-monitoring upgrade completes.
    The Helm upgrade does not always restart the pod automatically:

  kubectl rollout restart deployment/rancher-monitoring-kube-state-metrics \
    -n cattle-monitoring-system --request-timeout=30s

  kubectl rollout status deployment/rancher-monitoring-kube-state-metrics \
    -n cattle-monitoring-system --request-timeout=60s

4e. Verify metrics are flowing (allow ~60s after pod restart):

  # Query in Prometheus UI → Explore. kube-state-metrics adds kube_customresource_ prefix:
  kube_customresource_directpv_drive_total_capacity_bytes
  # Expected: one series per worker node, value 26843545600 (25 GiB per node)

  # If no results: check logs for config load confirmation:
  kubectl logs -n cattle-monitoring-system \
    -l app.kubernetes.io/name=kube-state-metrics --tail=20 --request-timeout=10s \
    | grep -i "directpv\|customresource"
  # Should show: "Adding metrics for ... directpv.min.io/v1beta1, Kind=DirectPVDrive"

4f. Verify the remoteWrite filter in Step 1 includes kube_customresource_directpv.*
    so these metrics are forwarded to VictoriaMetrics for 90-day trend panels.
    The regex already includes this pattern — confirm it is present:

  # In Rancher UI → Apps → rancher-monitoring → Edit YAML, look for:
  # regex: "...kube_customresource_directpv.*..."
